# شرح إصلاح مشكلة الـ Hot-Reload وفتح المنفذ في Nextron (Node v24+)

تم إجراء تعديلين أساسيين لضمان عمل بيئة التطوير بشكل مستقر وسلس:

## 1. مشكلة المنفذ المحجوز (EACCES: permission denied 0.0.0.0:8888)
**المشكلة:** نظام ويندوز (غالباً بسبب Hyper-V أو Docker) يحجز نطاقات من المنافذ (Ports) كاستخدام إداري، وكان المنفذ الافتراضي لـ Nextron (8888) يقع ضمن هذا النطاق المحجوز.
**الحل:** تم تغيير المنفذ الافتراضي إلى `1000` (أو أي منفذ متاح) في ملف `package.json`.

```json
"scripts": {
  "dev": "nextron --renderer-port 1000"
}
```

---

## 2. مشكلة انهيار السيرفر عند التعديل (SIGTERM Crash)
**المشكلة:** في الإصدارات الحديثة من Node.js (مثل الإصدار 24)، تقوم مكتبة `execa` (التي تعتمد عليها Nextron) برفض الوعد (Promise rejection) عند إغلاق عملية Electron القديمة باستخدام إشارة `SIGTERM` أثناء عملية الـ Hot-reload. هذا يؤدي إلى انهيار السيرفر بالكامل بدلاً من مجرد إعادة تشغيله.

**الحل التقني:**
تم تعديل ملف `node_modules/nextron/bin/nextron.cjs` لإضافة معالج أخطاء `.catch(()=>{})` عند تشغيل عملية Electron، مما يمنع الخطأ من إيقاف السيرفر.

**للإبقاء على هذا الإصلاح (Persistence):**
تم استخدام أداة تسمى `patch-package` لضمان عدم ضياع هذا التعديل عند حذف مجلد `node_modules` أو تثبيت الحزم مرة أخرى.

### الخطوات التي تم تنفيذها:
1. تثبيت `patch-package`: `npm install --save-dev patch-package`.
2. إنشاء رقعة برمجية (Patch): تم حفظ التعديل في مجلد `patches/nextron+10.0.0.patch`.
3. تحديث `package.json`: تمت إضافة `patch-package` إلى أمر الـ `postinstall` ليتم تطبيقه تلقائياً بعد كل عملية تثبيت للحزم.

```json
"scripts": {
  "postinstall": "patch-package && electron-builder install-app-deps"
}
```

---

## كيف تتعامل مع هذا مستقبلاً؟
- إذا قمت بتثبيت حزم جديدة (`npm install`)، سيقوم النظام تلقائياً بتطبق الرقعة البرمجية.
- يمكنك العثور على التفاصيل التقنية للرقعة في مجلد `patches`.
- إذا قمت بتحديث إصدار `nextron` لمستويات أعلى، قد تحتاج لإعادة إنشاء الرقعة إذا لم يقم المطورون بحل المشكلة في الإصدار الجديد.
